# Effectful Optics and Free Batching

### An idea from functional programming, made accessible to Java through Higher-Kinded-J

*If you have fought GraphQL N+1, written `$in` queries by hand, or shipped the half-updated immutable record, this piece shows how Higher-Kinded-J's optics let batching emerge from the data path itself — and how a small HKJ feature, coupled-field lenses, comes along for the ride.*

---

## The room and the stadium

Put everyone who already knows this technique in a room: Haskell users who reach for Haxl, `lens` people who run a traversal under a custom applicative. They will read what follows and nod. That room is small.

Now picture the stadium: the Java developers who have hand-wired `DataLoader`, hand-written `$in`, and shipped the half-updated-record bug. They have never been offered the declarative form. Not because it is hard, but because in their ecosystem it has not existed as usable infrastructure.

This piece is written for the stadium.

## Why Higher-Kinded-J

There are other functional libraries in Java; many feel like Haskell wearing a Java name tag, asking you to write pseudo-Haskell in a language that resists it. Higher-Kinded-J makes the opposite bet: it is **Java-first**, taking the good ideas from Haskell and FP and delivering them through Java's own modern grain — records, sealed interfaces, annotation processors, virtual threads, Spring. In this piece, that bet is visible end-to-end: the data is a record, the optics are generated by `@GenerateLenses`/`@GenerateFocus`, the async edge is virtual threads, the batch contract is shaped exactly like Java's own `MappedBatchLoader`, and the wiring is Spring. The functional idea arrives wearing Java clothes the stadium already owns.

That is the actual differentiator — best of both, not imitation of either — and the `hkj-book` makes the broader case if you want it.

## Two familiar bugs

**The N+1.** Load a `Team`, loop its 200 member ids, call `findById` 200 times. In GraphQL you reach for `DataLoader`; in Mongo you hand-write an `$in`. Either way, batching is a thing bolted next to your data access that every call site must remember to use.

**The half-updated record.** `Range(int lo, int hi)` with the invariant `lo <= hi`. Move `(1, 2)` to `(11, 12)` by setting `lo` first:

```
Range(1, 2) --set lo=11--> Range(11, 2)   // illegal, and it briefly existed
```

Two unrelated problems. They will pass through the same place below.

## What an optic is

Every Higher-Kinded-J optic reduces to one method:

```java
<F> Kind<F, T> modifyF(Function<A, Kind<F, B>> f, S s, Applicative<F> app);
```

In plain English: *reach into `s`, run `f` on each focused part, where `f` returns a value wrapped in some context `F`; use `app` to stitch the wrapped results back into a wrapped structure.* This is the standard van Laarhoven encoding. Its consequence is the only abstract idea you need:

> An optic is a **shape**. The `Applicative` is the **strategy**. The optic does not know which strategy it is running under.

HKJ already leans on this internally — `Traversal.asFold()` is the same `modifyF` run with the `Const` applicative, where the structural rebuild simply vanishes. Nothing below is more exotic than that.

## Swap the strategy, keep the path

One path, written once:

```java
Traversal<Company, Integer> salaries =
    CompanyTraversals.departments()
        .andThen(DepartmentTraversals.employees())
        .andThen(EmployeeLenses.salary().asTraversal());
```

| Applicative passed | What the same path does |
|---|---|
| identity | pure update |
| `Validated` | accumulate every error |
| `Const` + monoid | read-only fold (`asFold`) |
| concurrent | parallel traversal |
| `Fetch` | **batched load** |

The first four rows are uncontroversial HKJ. The fifth is the one this piece adds. It is the same move Haxl made in 2014, supplied where the others go — a sibling of `Const`, not a new framework.

## Coupled lenses

The half-update bug exists because a normal lens addresses one field at a time. HKJ's `Lens.paired` fuses two lenses into **one focus over a pair**:

```java
Lens<Range, Pair<Integer, Integer>> bounds = Lens.paired(loLens, hiLens, Range::new);

Range r = bounds.modify(p -> Pair.of(p.first() + 10, p.second() + 10), new Range(1, 2));
// Range(11, 12) — the reconstructor runs once, with both final values
```

The reconstructor — `Range::new` — is called once, with both final values. There is no in-between state because there is no in-between step. A coupled lens writes *"these fields move as a unit"* into the data model.

## What happens when a coupled lens rides the batching applicative

Because a coupled lens is still just a `Lens`, it flows through the same `modifyF`. Run it under the batching applicative:

- the pair's two field-loads are combined applicatively, so they land in the **same batch round**;
- the reconstructor still fires once, so the result is **rebuilt atomically and stays invariant-safe**.

The batching is a property of combining independent sibling loads applicatively — it happens for any siblings, coupled or not. The coupling contributes the atomic reconstruction. The two ride the same traversal, which is a tidy fact worth knowing: correctness and performance pass through one place, without you wiring them together.

## Batching as an applicative, in miniature

The essence — a value either finished or waiting on keys:

```
Fetch = Done(value) | Blocked(pendingKeys, resume)
```

- a single request: `Blocked({key}, results -> Done(results.get(key)))`;
- combine two independent fetches with `ap`: **union their pending keys** — one round;
- one fetch feeding the next with `flatMap`: the second's keys are unknowable until the first resolves — an extra round.

Run it: collect the pending keys, hand them to a single backend call, substitute the answers, repeat only if a `flatMap` forced another round. A small per-run cache removes repeat keys across rounds. Threaded through an optic traversal, the entire (deduplicated) path collapses to one round.

That is Haxl's idea, ported to a few hundred lines of Java against `modifyF`. No optic is modified. The whole thing is a new `Applicative` instance.

---

## The boundary, briefly

The batch contract is just `Set<K> -> CompletableFuture<Map<K, V>>`.

- **GraphQL:** that is `MappedBatchLoader` verbatim — the integration is near-identity.
- **MongoDB:** that `Set<K>` is one `{ _id: { $in: [...] } }`. No DataLoader infrastructure needed, and documents are naturally deep immutable trees that suit traversals.

The contract is generic by construction, so any keyed source fits. Mongo and GraphQL are two doors into the same room.

---

## Where it stops

A useful idea earns its keep by knowing where it doesn't apply.

**Applicative-only.** Everything here holds in *applicative* position. A real data dependency — `flatMap`, or `.via` on an effect path: "load the user, *then* load orders keyed by that user" — costs a round. The cache softens repeated keys; it cannot delete the round. Practical rule: model data dependencies as *shape* (deeper optics, coupled groups), not as *sequencing*.

**Not query pushdown.** This coalesces by-id lookups. It does not turn `.filtered(pred)` into a Mongo query or a GraphQL argument. Optics stay post-fetch.

**Open next steps.** Multi-source batching (several `(K, V)` families at once — HKJ's free applicative is the natural carrier); a railway error channel so a failed load composes as `Either`/`Validated`; coupled groups beyond pairs (N-ary, code-generated from a declared invariant); request-cache lifecycle for production wiring; a reactive (`Flux`/`Mono`) bridge; and the analyzability angle — because a `Blocked` node names its requests before running, the same seam could yield audit logs, dry-runs, and cost estimates for free.

---

## Closing

Higher-Kinded-J's `modifyF` seam lets you run an optic with a batching applicative and get request coalescing for free, without touching the optic. HKJ's coupled-field lens keeps its atomicity guarantee when it rides along. For a Java developer who has been wiring `DataLoader` by hand or writing `$in` queries one call site at a time, that is a markedly better way to spend the same problem — and it falls naturally out of the optic encoding HKJ already gives you.

Two doors. Same room. Java clothes.
